## 附录 C：技术栈选型详解

本附录对论文正文中提及的所有技术方案（含已选方案和备选方案）进行详细说明，包括其实现原理、解决需求、核心特性、技术思想，以及为什么适合或不适合 LearnAnything 项目。

### C.1 文档解析

#### C.1.1 MinerU CLI

**实现原理**：MinerU 是一个开源的文档解析工具，基于深度学习模型（LayoutLM 系列 + OCR）对 PDF 进行版面分析。其核心流程为：PDF 渲染 → 版面检测（识别文本块、图片、表格、公式区域）→ OCR/文本提取 → 结构化 Markdown 输出。

**解决需求**：将非结构化的 PDF 文档转化为结构化的 Markdown 文本，保留标题层级、图片位置、公式和表格等版面信息。

**核心特性**：
- 支持中文 PDF 的版面分析
- 输出标准 Markdown（含标题 `#` 标记、图片 `![]()`、公式 `$$`）
- 可识别表格并输出 Markdown 表格
- 支持 GPU 加速

**技术思想**：基于视觉的文档理解（Visual Document Understanding, VDU）——将文档解析视为计算机视觉任务，通过检测版面元素的位置和类型来重建文档结构。

**为什么适合本项目**：MinerU 在中文 PDF 的版面分析上准确率最高，能够正确识别标题层级（这是后续 MarkdownChunker 分块的基础），且开源可本地部署，无需外部服务。

**为什么不适合作为唯一方案**：MinerU 对扫描版 PDF 的 OCR 能力有限，且某些复杂版面的识别可能出错。因此本项目以 MinerU 为主，PyMuPDF 作为 fallback 处理扫描件。

#### C.1.2 PyMuPDF (fitz)

**实现原理**：PyMuPDF 是 MuPDF 的 Python 绑定，MuPDF 是一个轻量级的 PDF/EPUB/XPS 渲染引擎。PyMuPDF 提供低级 PDF 操作 API，包括页面渲染、文本提取、图片提取等。

**解决需求**：当 MinerU 无法正确解析（如扫描版 PDF、加密 PDF）时，作为 fallback 方案提取原始文本和图片。

**核心特性**：
- 速度极快（C 语言实现）
- 支持文本块的坐标提取（bbox）
- 支持图片直接提取（无需 OCR）
- 轻量，无深度学习依赖

**技术思想**：直接解析 PDF 的内部结构（内容流、字体、图片对象），不依赖视觉理解，因此速度快但结构信息有限。

**为什么适合本项目**：作为 MinerU 的 fallback，处理扫描版 PDF 时可以直接提取图片后交给 VLM 解析，无需等待 MinerU 的版面分析。

**为什么不适合作为主方案**：PyMuPDF 只能提取纯文本和图片，无法识别标题层级、表格结构、公式等版面信息，输出是非结构化的文本流。

#### C.1.3 PDFMiner

**实现原理**：PDFMiner 是一个纯 Python 的 PDF 解析库，通过解析 PDF 的内部对象树（Catalog → Pages → Content Streams）来提取文本和布局信息。

**解决需求**：从 PDF 中提取文本及其在页面上的精确位置（bbox），支持后续的布局分析。

**核心特性**：
- 纯 Python，跨平台
- 提供文本的精确坐标（bbox）
- 支持字体信息提取

**技术思想**：基于 PDF 规范（ISO 32000）的结构化解析，理解 PDF 的内部对象模型。

**为什么不适合本项目**：PDFMiner 仅提供文本和坐标，不内置版面分析（如标题识别、表格检测），需要额外的逻辑来重建文档结构。对于中文 PDF 的支持也不如 MinerU 完善。

#### C.1.4 Marker

**实现原理**：Marker 是另一个开源的 PDF 到 Markdown 转换工具，基于深度学习模型（如 Nougat）进行端到端的 PDF 解析。

**解决需求**：与 MinerU 类似，将 PDF 转化为结构化 Markdown。

**核心特性**：
- 基于端到端的深度学习模型
- 支持公式识别（LaTeX 输出）
- 支持表格识别

**技术思想**：端到端学习——训练一个模型直接从 PDF 图像输入到 Markdown 文本输出，无需显式的版面检测阶段。

**为什么不适合本项目**：Marker 的模型对中文 PDF 的支持不如 MinerU，且端到端模型的可解释性较差，当解析出错时难以调试和修复。此外，Marker 的部署依赖较大（需要下载大型模型权重）。

### C.2 分块策略

#### C.2.1 MarkdownChunker v2.0（本项目自研）

**实现原理**：基于 Markdown 的标题层级（`#` `##` `###`）和自然段落边界进行分块。具体流程：
1. 按 Markdown 标题层级建立树形结构
2. 每个自然段落（以换行分隔的文本块）作为一个 Paragraph Chunk
3. 每个标题作为一个 Heading Chunk
4. 每个文档作为一个 Document Chunk
5. 通过 `heading_path` 字段建立层级关系（如 `RAG概述 > RAG关键痛点`）
6. 通过 `BELONGS_TO` 边建立 chunk 之间的层级连接

**解决需求**：在保留文档结构（标题层级）的前提下，将文档切分为语义完整的自然段落，避免固定长度切分导致的语义断裂。

**核心特性**：
- 自然段是语义最小单元
- 树形 heading 层级保留文档结构
- 支持图片 chunk（`image_pseudo`）的标记
- 支持公式和表格的识别

**技术思想**：**语义完整性优先**——分块的基本单元应该是语义自洽的最小单元（自然段），而非固定长度的字符窗口。标题层级作为"语义容器"，帮助 LLM 理解段落的上下文。

**为什么适合本项目**：自然段是语义最小单元，不会切断句子；标题层级保留了文档的逻辑结构，为后续的 parent_hint 和 BELONGS_TO 关系提供了结构基础。

#### C.2.2 LangChain RecursiveCharacterTextSplitter

**实现原理**：按照预定义的分隔符列表（如 `\n\n`, `\n`, `. `, ` `）递归地切分文本，直到每个块的长度不超过 `chunk_size`。支持 `chunk_overlap`（块间重叠）以保持上下文连续性。

**解决需求**：将任意长文本切分为固定长度的块，便于 embedding 和向量检索。

**核心特性**：
- 可配置的分隔符列表
- 可配置的 chunk_size 和 chunk_overlap
- 支持按 token 数或字符数切分

**技术思想**：**固定窗口滑动**——将文本视为字符流，通过滑动窗口切分为等长的块，块间重叠以保持上下文。

**为什么不适合本项目**：固定长度切分会切断句子和段落，破坏语义完整性。例如，一个完整的论证过程可能被切分到两个块中，导致 embedding 无法捕获完整的语义。此外，RecursiveCharacterTextSplitter 不保留标题层级信息。

#### C.2.3 LlamaIndex SentenceSplitter

**实现原理**：基于句子的自然边界（句号、问号、感叹号）进行切分，保证每个块包含完整的句子。支持按句子数或 token 数控制块大小。

**解决需求**：在保持句子完整性的前提下，将文本切分为适当大小的块。

**核心特性**：
- 保证每个块以完整句子结束
- 支持按句子数或 token 数控制块大小
- 支持句子级别的重叠

**技术思想**：**句子完整性优先**——句子是自然语言的基本语义单位，保证句子完整性比分词或字符切分更合理。

**为什么不适合本项目**：虽然比固定长度切分更好，但 SentenceSplitter 仍然不保留标题层级信息。此外，技术文档中的"段落"往往比一个句子包含更完整的语义（一个段落通常是一个完整的论点或说明），SentenceSplitter 可能在句子级别切分过细。

### C.3 图片解析（VLM）

#### C.3.1 智谱 GLM-4V

**实现原理**：GLM-4V 是智谱 AI 开发的多模态大语言模型，基于 GLM（General Language Model）架构，支持视觉-语言联合理解。模型接收图片 + 文本提示，输出图片描述或回答。

**解决需求**：对文档中的图片（图表、示意图、截图）进行语义理解，生成文本描述，使图片内容可以被检索和概念提取。

**核心特性**：
- 中文场景下图片描述准确率高
- 支持多种图片类型（图表、流程图、截图、照片）
- API 调用成本低于 GPT-4V
- 支持图片中的文字 OCR

**技术思想**：**视觉-语言对齐**——通过大规模的图片-文本对预训练，学习视觉特征和语言特征的联合表示，使模型能够理解图片内容并用自然语言描述。

**为什么适合本项目**：智谱 GLM-4V 在中文场景下表现优秀（本项目主要处理中文文档），且 API 成本可控。生成的图片描述直接注入 chunk.text，使图片内容参与后续的 embedding 和概念提取。

#### C.3.2 GPT-4V (GPT-4 Turbo with Vision)

**实现原理**：GPT-4V 是 OpenAI 开发的多模态大语言模型，基于 Transformer 架构，支持图片输入。模型通过视觉编码器将图片转化为特征向量，与文本特征一起输入 Transformer 进行联合推理。

**解决需求**：同 GLM-4V，对图片进行语义理解和描述。

**核心特性**：
- 目前最强的通用多模态理解能力
- 支持复杂的视觉推理（如"图中 A 和 B 的关系是什么？"）
- 支持高清图片输入

**技术思想**：同 GLM-4V，视觉-语言对齐。

**为什么不适合本项目**：GPT-4V 的 API 成本较高（约为 GLM-4V 的 2-3 倍），且在国内访问需要代理。对于本项目的需求（生成图片描述），GLM-4V 的准确率已经足够，无需使用更昂贵的方案。

#### C.3.3 Qwen-VL

**实现原理**：Qwen-VL 是阿里云开发的多模态大语言模型，基于 Qwen（通义千问）架构，支持视觉-语言理解。

**解决需求**：同 GLM-4V。

**核心特性**：
- 开源模型，可本地部署
- 中文场景下表现良好
- 支持多种图片类型

**技术思想**：同 GLM-4V，视觉-语言对齐。

**为什么不适合本项目**：Qwen-VL 虽然开源可本地部署，但需要 GPU 资源（至少 16GB VRMB），而 LearnAnything 定位为桌面应用，用户可能没有足够的 GPU 资源。API 调用的方式（GLM-4V）更轻量。

### C.4 向量数据库

#### C.4.1 ChromaDB

**实现原理**：ChromaDB 是一个开源的嵌入式向量数据库，支持本地文件存储。其核心是一个基于 HNSW（Hierarchical Navigable Small World）算法的近似最近邻（ANN）索引，支持高效的向量相似度搜索。

**解决需求**：存储文档 chunk 的 embedding 向量，支持基于语义相似度的快速检索。

**核心特性**：
- 本地文件存储（SQLite + Parquet），无需外部服务
- 支持 HNSW 索引（O(log n) 的搜索复杂度）
- 支持元数据过滤（metadata filtering）
- 支持多种距离度量（cosine, euclidean, inner product）
- 简单的 Python API

**技术思想**：**嵌入式数据库**——向量数据库直接嵌入到应用程序中，无需独立的服务器进程，简化了部署和运维。

**为什么适合本项目**：ChromaDB 的本地文件存储模式非常适合桌面应用——用户无需配置外部数据库服务，数据保存在本地，保护隐私。HNSW 索引保证了大规模数据下的检索效率。

#### C.4.2 Pinecone

**实现原理**：Pinecone 是一个托管的向量数据库服务，提供全托管的向量索引和搜索 API。用户通过 HTTP API 上传向量和元数据，Pinecone 负责索引构建和查询优化。

**解决需求**：同 ChromaDB，但提供企业级的可用性和扩展性。

**核心特性**：
- 全托管，无需运维
- 自动扩展
- 支持混合搜索（向量 + 元数据过滤）
- 提供 SOC2 合规和 SLA 保障

**技术思想**：**数据库即服务（DBaaS）**——将向量索引和搜索的复杂性交给云服务提供商，用户只需关注业务逻辑。

**为什么不适合本项目**：Pinecone 是云服务，需要网络连接和 API Key，且按使用量计费。LearnAnything 定位为本地桌面应用，用户可能在没有网络的环境下使用，且不希望数据上传到云端。此外，Pinecone 的成本对于个人用户来说较高。

#### C.4.3 Weaviate

**实现原理**：Weaviate 是一个开源的向量搜索引擎，支持向量和结构化数据的混合查询。基于 HNSW 索引，支持 GraphQL 查询接口。

**解决需求**：同 ChromaDB，但提供更丰富的查询能力（混合搜索、GraphQL 接口）。

**核心特性**：
- 开源，可本地部署
- 支持 GraphQL 查询
- 支持模块化 AI 集成（可接入各种 embedding 模型）

**技术思想**：**向量 + 图数据库的融合**——Weaviate 不仅支持向量搜索，还支持对象之间的关系查询，类似于轻量级的图数据库。

**为什么不适合本项目**：Weaviate 的功能比 ChromaDB 更丰富，但也更复杂。LearnAnything 的向量检索需求相对简单（主要是语义相似度搜索），ChromaDB 的轻量 API 足够满足需求，且更易于集成。

#### C.4.4 Milvus

**实现原理**：Milvus 是一个开源的分布式向量数据库，专为大规模向量数据设计。支持多种索引类型（HNSW, IVF, ANNOY 等），支持分布式部署和水平扩展。

**解决需求**：超大规模向量数据的存储和检索（十亿级别）。

**核心特性**：
- 分布式架构，支持水平扩展
- 多种索引类型可选
- 支持 GPU 加速
- 企业级功能（RBAC、多租户）

**技术思想**：**分布式向量搜索**——将向量索引分片存储在多个节点上，通过分布式查询协调器并行搜索，实现超大规模数据的高效检索。

**为什么不适合本项目**：Milvus 的分布式架构对于桌面应用来说是过度设计。LearnAnything 的单学科数据量通常在数千到数万个向量，ChromaDB 的本地 HNSW 索引完全足够。Milvus 的部署和维护成本对于个人用户来说过高。

### C.5 图数据库

#### C.5.1 KùzuDB

**实现原理**：KùzuDB 是一个嵌入式图数据库管理系统（Graph DBMS），采用列式存储和向量化查询执行引擎。支持 Cypher 查询语言，与 Neo4j 兼容。KùzuDB 将图数据存储在本地文件中，通过内存映射实现高效访问。

**解决需求**：存储和查询双层知识图谱（Chunk 层 + Concept 层），支持复杂的图遍历和关系查询。

**核心特性**：
- 嵌入式，单文件存储（类似 SQLite）
- Cypher 查询语言兼容（与 Neo4j 语法基本一致）
- 列式存储 + 向量化执行（查询效率高）
- 支持 ACID 事务
- 支持多种数据类型（包括 LIST、STRUCT）

**技术思想**：**嵌入式图数据库**——将图数据库引擎嵌入到应用程序中，无需独立的服务器进程，数据以文件形式存储在本地，兼顾了图数据库的强大查询能力和嵌入式数据库的易用性。

**为什么适合本项目**：KùzuDB 的嵌入式特性完美契合桌面应用的需求——无需安装和配置数据库服务器，数据保存在本地文件中，跨平台兼容。Cypher 查询语言的兼容性使得开发者可以利用 Neo4j 的生态资源。

#### C.5.2 Neo4j

**实现原理**：Neo4j 是最流行的图数据库，基于属性图模型（Property Graph Model），使用原生图存储引擎（Native Graph Storage）。数据以节点和关系的物理形式存储，而非关系型数据库的表结构。

**解决需求**：企业级图数据存储和查询，支持复杂的图算法和社区检测。

**核心特性**：
- 原生图存储（非关系型后端模拟）
- 强大的 Cypher 查询语言
- 丰富的图算法库（PageRank、社区检测、最短路径等）
- 企业级功能（集群、备份、安全）

**技术思想**：**原生图存储**——节点和关系在磁盘上以物理指针的形式存储，图遍历查询的时间复杂度与路径长度成正比（O(k)，k 为跳数），而非关系型数据库的 O(n) 表扫描。

**为什么不适合本项目**：Neo4j 需要独立的服务器进程（Java 运行），部署复杂，资源占用高。对于桌面应用来说，用户需要额外安装和配置 Neo4j，这与 LearnAnything"开箱即用"的设计理念不符。此外，Neo4j 的社区版功能有限，企业版需要付费。

#### C.5.3 Dgraph

**实现原理**：Dgraph 是一个开源的分布式图数据库，采用原生图存储和分布式架构。支持 GraphQL+- 查询语言（扩展的 GraphQL），支持水平扩展。

**解决需求**：大规模分布式图数据的存储和查询。

**核心特性**：
- 分布式架构，支持水平扩展
- 原生图存储
- 支持 GraphQL+- 查询

**技术思想**：**分布式原生图数据库**——在原生图存储的基础上增加分布式能力，通过分片（sharding）和复制（replication）实现水平扩展。

**为什么不适合本项目**：Dgraph 的分布式架构对于桌面应用来说是过度设计，且部署和维护成本较高。查询语言 GraphQL+- 与 Cypher 不同，学习成本较高。

### C.6 Embedding 模型

#### C.6.1 智谱 AI Embedding API

**实现原理**：智谱 AI 的文本嵌入模型基于 Transformer 架构（类似 BERT），通过大规模中文语料的预训练，学习文本的语义表示。输入文本经过 tokenization → Transformer 编码 → 池化（pooling）→ 输出 2048 维的稠密向量。

**解决需求**：将文本（chunk、concept、query）转化为稠密向量，支持语义相似度计算和向量检索。

**核心特性**：
- 2048 维向量（高维度，信息容量大）
- 中文语义效果好（基于大规模中文语料预训练）
- 支持长文本（最长 8192 token）
- API 调用成本可控

**技术思想**：**对比学习（Contrastive Learning）**——通过训练模型使得语义相似的文本在向量空间中距离近，语义不相似的文本距离远。

**为什么适合本项目**：中文场景下表现优秀，2048 维向量提供了足够的信息容量来区分细粒度的语义差异，且 API 成本可控。

#### C.6.2 OpenAI text-embedding-3

**实现原理**：OpenAI 的 text-embedding-3 系列模型基于 Transformer 架构，通过大规模多语言语料的预训练生成文本嵌入。支持 `text-embedding-3-small`（512/1536 维）和 `text-embedding-3-large`（256/3072 维）。

**解决需求**：同智谱 Embedding，但支持多语言场景。

**核心特性**：
- 多语言支持（不仅限于中文）
- 支持多种维度（可压缩到 256 维以节省存储）
- 性能优秀（在 MTEB 排行榜上排名靠前）

**技术思想**：同智谱 Embedding，对比学习。

**为什么不适合本项目**：OpenAI 的 API 在国内访问需要代理，且成本较高。对于主要处理中文文档的 LearnAnything 来说，智谱 Embedding 在中文场景下表现相当，且更易访问。

**核心特性**：
- 开源，可本地部署

- 支持多种尺寸（base/large）

### C.9 布局算法

#### C.9.1 dagre（Directed Graph Renderer）

**实现原理**：dagre 是一个用于有向图层次布局的 JavaScript 库，基于 Gansner 等人的算法（GKNV 算法）。核心流程：
1. 去环（Cycle Removal）：通过反向边消除图中的环
2. 分层（Layer Assignment）：将节点分配到不同的层级（rank），最小化边跨越的层数
3. 顶点排序（Vertex Ordering）：在每个层级内排序节点，最小化边交叉
4. 坐标分配（Coordinate Assignment）：为每个节点分配 x/y 坐标

**解决需求**：为树形和 DAG 数据提供清晰的层次布局（上层在左，下层在右，或上层在上，下层在下）。

**核心特性**：
- 清晰的层次结构（rank）
- 最小化边交叉
- 支持多种方向（LR、RL、TB、BT）
- 支持边的虚拟节点（处理长跨层边）

**技术思想**：**层次化布局**——通过将节点分层和排序，使得图的结构清晰可读，特别适用于表达层级关系（如组织架构、依赖关系、文档结构）。

**为什么适合本项目**：dagre 的层次布局非常适合文档树（document → heading → paragraph 的层级）和概念图谱（核心概念 → 子概念 → 实现技术的层级）。LR（从左到右）方向使得根节点在左侧，子节点在右侧，符合阅读习惯。

#### C.9.2 COSE（Compound Spring Embedder）

**实现原理**：COSE 是 Cytoscape.js 内置的力导向布局算法，基于弹簧-电荷物理模型。节点之间像弹簧一样互相吸引（边的张力），同时像电荷一样互相排斥（防止节点重叠）。通过迭代模拟物理系统的能量最小化过程，最终达到平衡状态。

**解决需求**：为网络图（无明确层级的图）提供美观的布局。

**核心特性**：
- 力导向布局（物理模拟）
- 支持复合图（节点可以包含子图）
- 自适应（自动调整节点间距）

**技术思想**：**物理模拟**——将图视为物理系统，节点是带电粒子，边是弹簧，通过模拟物理系统的演化达到美观的布局。

**为什么不适合本项目**：力导向布局缺乏明确的层次结构，对于表达"文档层级"或"概念依赖关系"不够清晰。dagre 的层次布局更适合本项目的树形/DAG 数据。

### C.10 Agent 框架

#### C.10.1 自研 Agent 框架（Coordinator + MessageBus）

**实现原理**：LearnAnything 采用自研的轻量级 Agent 框架：
- **Coordinator**：中央调度器，负责意图路由、上下文组装、Agent 分发
- **MessageBus**：发布-订阅消息总线，支持 Agent 间的异步通信
- **BaseAgent**：所有 Agent 的基类，定义统一的接口（handle、setup、teardown）
- **IntentRouter**：基于正则和关键词的意图识别模块

**解决需求**：实现 QuizAgent、CoachAgent、TutorAgent 的协作，支持从出题到测评到讲解的完整学习闭环。

**核心特性**：
- 轻量（无外部依赖）
- 去中心化（Agent 通过消息总线通信，非直接调用）
- 可扩展（新增 Agent 只需注册到 Coordinator）
- 审计（所有消息通过 MessageBus 记录，支持追溯）

**技术思想**：**事件驱动架构**——Agent 之间不直接耦合，而是通过事件（消息）进行通信，每个 Agent 订阅自己关心的事件，实现松耦合和可扩展性。

**为什么适合本项目**：自研框架轻量且完全可控，不需要引入 LangChain 等重型依赖。去中心化的设计使得新增 Agent（如 Headhunter）非常容易。

#### C.10.2 LangChain

**实现原理**：LangChain 是一个用于构建 LLM 应用的 Python/JavaScript 框架，提供了一系列组件（Chains、Agents、Memory、Retrievers）来简化 LLM 应用的开发。

**解决需求**：快速构建基于 LLM 的应用，包括 RAG、Agent、工具调用等。

**核心特性**：
- 丰富的组件库（Chains、Agents、Memory、Vector Stores 等）
- 支持多种 LLM 和 Vector Store 的抽象接口

**技术思想**：**组件化 LLM 应用开发**——将 LLM 应用的常见模式（RAG、ReAct Agent 等）封装为可复用的组件，开发者通过组合组件快速构建应用。

**为什么不适合本项目**：LangChain 的抽象层次较高，对于本项目的特定需求（双层知识图谱、多范式概念提取、图状态知识追踪），LangChain 的标准组件无法直接满足，需要大量自定义。此外，LangChain 的版本更新频繁，API 变化较大，维护成本高。

### C.11 知识追踪

#### C.11.1 基于图的状态传播（本项目自研）

**实现原理**：LearnAnything 的知识追踪不依赖深度学习模型，而是基于知识图谱的拓扑结构进行状态传播：
1. 每个 CanonicalConcept 对应一个用户掌握度状态（mastery_level, confidence, theta）
2. 当用户答对/答错某概念时，直接更新该概念的状态
3. 沿 DEPENDS_ON 边向依赖概念传播（掌握度提升 → 前置概念置信度提升）
4. 沿 SOLUTION 边向父概念传播（子概念掌握 → 父概念掌握度提升）

**解决需求**：在没有大量历史数据的情况下，实现自适应测评和知识状态追踪。

**核心特性**：
- 无需预训练数据
- 可解释（状态更新路径清晰）
- 与知识图谱结构天然结合
- 实时更新

**技术思想**：**拓扑传播**——利用知识图谱的结构信息（概念间的依赖关系）来推断用户的知识状态，而非依赖历史答题数据训练模型。

**为什么适合本项目**：LearnAnything 在初期没有大量用户答题数据，无法训练 DKT 等深度学习模型。基于图的状态传播不需要预训练，直接利用知识图谱的结构，适合冷启动阶段。

#### C.11.2 DKT（Deep Knowledge Tracing）

**实现原理**：DKT 是基于循环神经网络（RNN/LSTM）的知识追踪模型。将学生的答题历史序列输入 LSTM，预测下一个题目的答对概率。模型学习每个知识点的隐状态表示。

**解决需求**：基于大量历史答题数据，预测学生的知识掌握状态。

**核心特性**：
- 基于深度学习，可以捕捉复杂的答题模式
- 支持知识点的隐状态表示
- 需要大量训练数据（通常 >10000 条答题记录）

**技术思想**：**序列建模**——将学生的答题历史视为时间序列，通过 RNN 建模序列中的时序依赖关系。

**为什么不适合本项目**：DKT 需要大量的历史答题数据（至少每个知识点数十次答题记录）才能达到可靠的效果。LearnAnything 在初期用户量有限，无法提供足够的训练数据。此外，DKT 是黑盒模型，状态更新路径不可解释。

### C.12 信息检索

#### C.12.1 BM25

**实现原理**：BM25（Best Match 25）是一种基于概率检索框架的稀疏检索算法。核心思想是：词项在文档中的出现频率（TF）和逆文档频率（IDF）共同决定文档的相关性得分。BM25 对 TF 进行饱和处理（避免高频词主导），并考虑文档长度的归一化。

公式：`score(q, d) = Σ IDF(qi) * (f(qi, d) * (k1 + 1)) / (f(qi, d) + k1 * (1 - b + b * |d| / avgdl))`

**解决需求**：基于关键词的精确检索，快速找到包含查询词的文档。

**核心特性**：
- 基于词的精确匹配（非语义）
- 速度快（倒排索引，O(1) 查询）
- 可解释性强（每个词的得分独立计算）
- 无需预训练

**技术思想**：**概率检索模型**——将检索视为概率估计问题，计算文档与查询相关的概率。

**为什么适合本项目**：BM25 作为混合检索的第一阶段，可以快速筛选出包含关键词的候选文档，为后续的向量重排序缩小范围。其可解释性也便于调试和优化。

#### C.12.2 RRF（Reciprocal Rank Fusion）

**实现原理**：RRF 是一种结果融合算法，用于合并多个检索系统（如 BM25 和向量检索）的结果列表。对于每个文档，RRF 计算其在各个列表中的排名的倒数之和：`score(d) = Σ 1 / (k + rank_i(d))`，其中 k 是一个常数（通常取 60），用于防止排名靠前的文档得分过于主导。

**解决需求**：融合多个检索策略（稀疏 + 密集）的结果，平衡精确性和语义覆盖。

**核心特性**：
- 无需训练参数（k=60 是经验值）
- 对排名敏感，对绝对得分不敏感
- 简单高效

**技术思想**：**排名融合**——不同检索系统的得分尺度不同，直接加权求和不公平。RRF 将绝对得分转化为排名，通过排名的倒数融合，避免某一系统的主导。

**为什么适合本项目**：RRF 简单有效，不需要训练数据或参数调优。对于 BM25（基于排名）和向量检索（基于相似度得分）的融合，RRF 是一种公平且鲁棒的方法。

#### C.12.3 HNSW（Hierarchical Navigable Small World）

**实现原理**：HNSW 是一种用于近似最近邻（ANN）搜索的图索引算法。通过构建多层图结构（类似跳表），在底层保持精确的邻接关系，在高层建立"捷径"连接。查询时从顶层开始，通过贪心算法快速接近目标区域，然后在底层精确搜索。

**解决需求**：在高维向量空间中快速找到与查询向量最相似的向量。

**核心特性**：
- 搜索复杂度 O(log n)
- 构建复杂度 O(n log n)
- 支持动态插入和删除
- 召回率高（通常 >95%）

**技术思想**：**多层图导航**——通过构建分层的近似图，利用"小世界"现象（任何两个节点之间都可以通过少量跳数到达），实现高效的最近邻搜索。

**为什么适合本项目**：HNSW 是 ChromaDB 的默认索引算法，为本项目的向量检索提供了高效的底层支持。O(log n) 的搜索复杂度使得即使在大规模数据下（数万向量），检索仍然可以在毫秒级完成。

### C.13 IRT（项目反应理论）

#### C.13.1 单参数 Rasch 模型

**实现原理**：Rasch 模型是 IRT 中最简单的模型，假设每个题目只有一个参数——难度（b）。学生的能力（θ）和题目的难度（b）共同决定答对概率：`P(θ) = 1 / (1 + exp(-(θ - b)))`。当学生能力高于题目难度时，答对概率 > 0.5。

**解决需求**：在没有大量数据的情况下，实现基本的自适应测评（根据学生能力选择合适难度的题目）。

**核心特性**：
- 参数少（只有难度 b），容易估计
- 可解释性强（b 就是题目的难度值）
- 适合小样本（几十到几百人次即可估计）

**技术思想**：**概率建模**——将答题结果建模为能力和难度的概率函数，通过最大似然估计（MLE）或贝叶斯估计（EAP）来估计学生能力。

**为什么适合本项目**：Rasch 模型参数少，适合 LearnAnything 初期的数据量（每个题目可能只有几十次答题记录）。其可解释性也便于调试和向用户解释"为什么推荐这道题"。

#### C.13.2 三参数 IRT（3PL）

**实现原理**：三参数 IRT 模型在 Rasch 的基础上增加了两个参数：区分度（a）和猜测参数（c）。答对概率公式为：`P(θ) = c + (1 - c) / (1 + exp(-a * (θ - b)))`。区分度 a 表示题目区分高能力和低能力学生的能力，猜测参数 c 表示随机猜对的概率。

**解决需求**：更精确地建模题目的统计特性，特别是在大规模测评场景中。

**核心特性**：
- 参数多（a, b, c），模型更灵活
- 可以建模猜测行为（如选择题的随机猜测）
- 需要大量数据（每个题目至少 100-200 人次才能可靠估计）

**技术思想**：同 Rasch，概率建模，但增加了更多参数来捕捉题目的复杂性。

**为什么不适合本项目**：3PL 模型需要每个题目有大量的答题记录（通常 >100 人次）才能可靠估计参数。LearnAnything 在初期无法提供足够的数据。此外，3PL 的参数估计更复杂，可能出现收敛问题。
