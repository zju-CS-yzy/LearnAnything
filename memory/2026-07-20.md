### 6. LA-027: 范式 v2.0 修正（DEPEND_ON 语义纠正）

**开发者审阅反馈**：
`technology --DEPEND_ON--> requirement` 不是"反向引用边"（会导致图谱环），而是**技术分解出的子需求**（DAG 结构）。

**正确理解**：
```
requirement(顶层驱动) --IMPLEMENTS--> technology
    ↑                                    │
    └────────── DEPEND_ON ──────────────┘
         （技术分解出的子需求/约束）

示例：
"为了提升训练效率，采用多GPU并行训练。多GPU并行训练需要保证梯度同步正确。"
→ "提升训练效率" --IMPLEMENTS--> "多GPU并行训练"
→ "多GPU并行训练" --DEPEND_ON--> "保证梯度同步正确"
```

**关键修正**：
1. requirement 既可以是顶层驱动（无 parent），也可以是 technology 的子节点（子需求）
2. parent_rules: requirement: [technology]（允许子需求）
3. DEPEND_ON 的语义："技术分解出的子需求/约束"，不是"反向引用"
4. prompt_addon 必须清晰描述两种关系的语义差异（已重写）
5. relation 语义描述应暴露给用户填写（前端表单中增加 relation 语义说明字段）

**修改文件**：
- `config/paradigms.yaml`：engineering 的 relation_map、parent_rules、ideal_chain、prompt_addon 全部重写
- `docs/DESIGN.md`：14.2 节删除"反向引用边"术语，改为 DAG 结构描述

**最终 engineering 结构**：
```
types:      requirement, technology
relations:  IMPLEMENTS, DEPEND_ON

IMPLEMENTS: "需求驱动技术"（"为了解决X，采用Y"）
DEPEND_ON:  "技术分解出子需求"（"Y需要满足Z"、"Y的前提是Z"）

relation_map:
  requirement --IMPLEMENTS--> technology
  technology --DEPEND_ON--> requirement

parent_rules:
  requirement: [technology]   # 子需求的 parent 是技术
  technology: [requirement]   # 技术的 parent 是顶层需求
```
