### 1. P27 排查：前端图片显示失败

**用户现象**：点击有 `图片×1` 标签的节点，NodeDetailPanel 不显示图片，Network 中无图片请求。

**排查链路**：
1. 读取前端代码：GraphView.vue / NodeDetailPanel.vue / GraphNodeTooltip.vue
   - `NodeDetailPanel` 图片 URL 构造逻辑正确：`/api/images/{subject}/{filename}`
   - `hasMedia` 条件：`media_refs?.length > 0`
2. 写诊断脚本（多次迭代，处理编码问题）
3. KùzuDB 数据检查：
   - `rag_v1`：45/756 概念节点有图片（6%）
   - `transformer_v1`：27/139 概念节点有图片（19%）
4. **用户截图分析**：节点 ID 为 `concept_canonical_b9dcd94454`（"文本生成"）
5. 该节点 `media_refs` 为空，fallback 从 `source_chunks` 获取也失败

**根因**：
- Chunk 节点 `media_refs` 存储了 Windows 绝对路径：`D:\MyCS\...`
- JSON 中反斜杠未转义 → `json.loads` 失败：`Invalid \escape`
- `_get_media_refs_from_chunks` fallback 解析失败 → 概念节点 `media_refs` 为空
- 前端 `hasMedia` 为 false → 不渲染 `img` → Network 无图片请求

### 5. P30 排查：文档树卡死

**用户观察**：CanonicalConcept 树（756 节点）不卡死，但文档树（454 chunk 节点）卡死，说明不是数量问题。

**排查链路**：
1. 读取 `GraphView.vue` / `GraphLayout.js`：`runTreeLayout` 有 >200 节点保护机制
2. 写诊断脚本 `analyze_tree_structure.py` 分析数据库结构
3. **关键发现**：BELONGS_TO / ADJACENT_TO 结构完全混乱：
   - BELONGS_TO: 816 条 paragraph→paragraph（应为 ADJACENT_TO），82 条反向 paragraph→heading，102 条反向 image_pseudo→paragraph
   - ADJACENT_TO: 551 条 heading→paragraph（层级关系混入同级顺序），113 条 heading→heading
   - document 节点无出边（5 个叶子），167 个 heading 无父节点
   - 根节点：167 heading + 21 paragraph（伪根）

**根因**：`build_belongs_to_relations` 按 heading_path 分组后，组内所有相邻 chunk 都建 BELONGS_TO（不区分类型）；`build_adjacent_relations` 按全局文档排序建边（跨 heading）；document 未连接 heading。

**修复**（后端数据重构）：
1. 删除旧边：DELETE 1527 BELONGS_TO + 2221 ADJACENT_TO（KuzuDB 支持 DELETE 关系）
2. 重建正确结构：
   - document→heading: 82（5 文档 → 82 顶层标题）
   - heading→heading: 101（数字编码前缀匹配 + `>` 路径分隔）
   - heading→paragraph: 356（heading_path 匹配）
   - heading→image_pseudo: 25
   - paragraph→paragraph: 215（同级顺序）
   - image_pseudo→image_pseudo: 1

**代码修复**（`core/graph_store.py`）：
- 重写 `build_belongs_to_relations`：加载所有 chunks，建立 heading_by_path / heading_by_code 映射，提取数字编码和 `>` 分隔层级
- 重写 `build_adjacent_relations`：按 (source, heading_path, type) 分组，仅同级 chunk 建边
- `build_has_parent_relations`：标记为已合并到 BELONGS_TO

### 5. P0-QUIZ 修复：Agent 出题流程回退

**问题回顾**：前端调用出题接口时，P0 流程（ConceptRetriever → SubgraphBuilder → ContextAssembler）未执行，回退到旧方式。

**排查链路**：
1. 读取 `backend_api.py`：发现 `/api/quiz` 和 `/api/evaluate/start` 直接调用 `QuizAgent`/`CoachAgent`，**完全绕过 Coordinator**
2. 读取 `coordinator.py`：P0 流程在 `Coordinator.handle()` 中已实现，但出题端点从未调用它
3. 检查 KuzuDB 连接：`backend_api.py` 中每个请求独立创建 `GraphStore`（多处 `GraphStore(f"{subject}_v1")`），每个实例打开独立的 KuzuDB 连接
4. 用户猜测验证：前端知识图谱查询 + 出题并发时，多个 GraphStore 实例打开同一个 KuzuDB 文件 → 文件锁定冲突

**根因**：
1. **API 端点绕过 Coordinator**：`/api/quiz` 直接 `QuizAgent.handle()`，`/api/evaluate/start` 直接 `CoachAgent.handle()`，P0 模块（ConceptRetriever/SubgraphBuilder/ContextAssembler）完全未使用
2. **KuzuDB 文件锁定**：每个请求独立创建 GraphStore，并发时多个连接打开同一文件

**修复**：
1. `backend_api.py` 全局 GraphStore 缓存：`_graph_store_cache = {}`，`get_graph_store(subject)` 每个学科只创建一个连接
2. `/api/quiz` 改用 `Coordinator` + 共享 GraphStore
3. `/api/evaluate/start` (mixed/default 模式) 改用 `Coordinator` + 共享 GraphStore
4. `Coordinator.__init__` 支持外部传入 `graph_store` 参数
